# Cloudless Provider — full-surface + hardening design

Status: draft, awaiting review.
Date: 2026-05-03.
Repo: `cloudless-terraform` (Terraform provider for the Fluence public API).

## Goal

Take the existing thin-slice provider (4 resources, 3 data sources, no tests) and
ship a complete, hardened provider:

- Cover the rest of the Fluence resource surface — security groups, storage,
  public IPs, plus two attachment-style companion resources
  (`cloudless_vm_public_ip_attachment`, `cloudless_security_group_attachment`).
- Add a polished cluster-discovery story with country/city filters.
- Add a thin convenience for subnets — derive `cluster_id` from the parent
  VPC when not supplied. All other cluster-scoped resources keep an explicit
  `cluster_id`. (Provider-level default was considered and rejected: see D6.)
- Land unit tests (TDD-driven), schema-level validators, generated docs, a
  release pipeline (`goreleaser`), and CI.

The work splits into two implementation plans (Phase 1, Phase 2). This single
spec covers both.

## Non-goals

- No `cloudless_region` resource and no rename of `cluster_id` to `region`.
  Fluence's locality is per-cluster; calling it anything else hides the
  constraint.
- No automatic cluster selection ("auto-pick a cluster in this region"). All
  cluster choices remain deterministic.
- No CI-gated acceptance tests in this spec. Acceptance tests are written and
  runnable locally (`TF_ACC=1`); turning them on in CI is its own milestone.
- No first published release. The release pipeline (goreleaser, GPG signing) is
  configured but not triggered. Tagging the first release is a separate step.

## Decisions agreed during brainstorming

| # | Decision | Source |
|---|---|---|
| D1 | Resource prefix is `cloudless_` even though API is Fluence | project memory `project_naming.md` |
| D2 | Each VM relationship uses the model that fits its API shape — see "Resource relationships" table below | revised during entity walkthrough |
| D3 | Security group rules: per-direction `ingress_mode` / `egress_mode` enum (`allow_all` (default) / `allow_listed` / `deny_all`) plus repeated `ingress {}` / `egress {}` blocks. Mode controls how blocks are interpreted; mutual exclusion is structural. | Q4-C, revised twice (deny-all gap, then enum for symmetry with allow-all) |
| D4 | Acceptance tests live in repo, opt-in via `TF_ACC=1`, never in CI | Q5-B |
| D5 | Phase 1 = new resources with TDD unit tests. Phase 2 = cross-cutting hardening | Q6-C |
| D6 | `cluster_id` is **Required** on every cluster-scoped resource. **No** provider-level default. Subnet additionally derives from VPC if not explicitly set. | revised — user dislikes provider-level default-cluster magic |
| D7 | "Region" is an ISO 3166-1 alpha-2 country code. It is a *filter* attribute on data sources only — never a resource attribute. | latest exchange |
| D8 | New data sources: `cloudless_cluster` (singular, errors on ambiguity) and enhanced `cloudless_clusters` (filterable list). Both join datacenter info and expose `region`, `city_code`, `dc_id`, `dc_slug`, `dc_tier` as computed fields | latest exchange |
| D9 | Datacenter is computed-only output; never user input | latest exchange |
| D10 | SG rule `type` (ipv4/ipv6) defaults to `"ipv4"` | recommendation accepted via "do not ask too" |
| D11 | `cloudless_storage.volume_gb` is in-place updatable via PATCH (matches API), with documented warning about VM restart | recommendation accepted via "do not ask too" |

### Layering

```
HCL author
  ↓
terraform-plugin-framework  (schema, plan, lifecycle hooks)
  ↓
internal/provider/*          (resource definitions, type conversion)
  ↓
internal/client/*            (HTTP, JSON, polling, error mapping)
  ↓
Fluence API
```

`internal/client/` knows nothing about Terraform types. It returns plain Go
structs and a typed `*APIError`. `internal/provider/` is the only layer that
imports plugin-framework symbols. The mock server in `internal/client/mock/`
sits at the same layer as the real API and implements the subset of CRUD the
unit tests exercise.

## Resource relationships (the model behind D2)

Different relationships use different patterns based on what the API supports
*today* and what we expect *tomorrow*:

| Relationship | Where it lives | Why |
|---|---|---|
| VM → boot disk | Inline `boot_disk { storage_id \| ...inline create }` block on `cloudless_vm`, ForceNew | Boot-disk swap recreates the VM by definition; no day-2 rebind |
| VM → data disks | `data_disk_ids = [...]` list on `cloudless_vm`, smart Update via `/storages/add` and `/storages/remove` | API supports N today; no future-proofing argument; ID list is shorter HCL |
| VM → SSH keys | `ssh_key_ids = [...]` list on `cloudless_vm`, ForceNew | Fluence applies SSH keys at first boot only; can't change post-create |
| VM → public IP | Separate resource `cloudless_vm_public_ip_attachment` | Fluence may add multi-IP-per-VM later; attachment resource avoids a future breaking schema change |
| VM → security group | Separate resource `cloudless_security_group_attachment` keyed on `network_interface_id` | SG is interface-scoped per the API; a VM-level field would be a fiction. Multi-interface VMs are possible. |

There is no `cloudless_vm_storage_attachment`. Considered, dropped.

## Provider configuration

```hcl
provider "cloudless" {
  api_key  = var.fluence_api_key                # or FLUENCE_API_KEY
  endpoint = "https://api.fluence.dev"          # optional override; this is the default
}
```

| Attribute | Required | Env fallback | Notes |
|---|---|---|---|
| `api_key` | yes (or env) | `FLUENCE_API_KEY` | sensitive |
| `endpoint` | no | `FLUENCE_ENDPOINT` | defaults to `https://api.fluence.dev` |

There is **no `default_cluster_id`**. Cluster scoping is per-resource. See D6
in the decisions table for the rationale (kept simple/explicit; multi-cluster
configs use distinct `data.cloudless_cluster` blocks per cluster, not provider
aliases).

## Data sources

### `cloudless_cluster` (NEW — singular picker)

```hcl
data "cloudless_cluster" "main" {
  region    = "DE"        # ISO 3166-1 alpha-2 country code; optional
  city_code = "FRA"       # IATA-style city code; optional
  name      = "..."       # cluster human name; optional
  id        = "..."       # explicit cluster UUID; optional
}
```

Behavior: filter all clusters by every supplied criterion. After filtering,
**exactly one cluster must match**, else error. Composition is "AND".

Computed outputs: `id`, `name`, `region`, `city_code`, `dc_id`, `dc_slug`,
`dc_tier`, `dc_certifications`.

### `cloudless_clusters` (enhanced — list)

```hcl
data "cloudless_clusters" "eu" {
  regions    = ["DE", "PL"]   # optional
  city_codes = ["FRA", "WAW"] # optional
  names      = ["Cloudless"]  # optional
}
```

Returns `clusters` list with the same per-element shape as `cloudless_cluster`.
Empty list is a valid result.

Both data sources transparently issue `/v1/datacenters` and `/v1/clusters`,
join in memory, then filter. They cache the join inside one Terraform run so
multiple data blocks don't refetch.

### `cloudless_vm_configurations`, `cloudless_default_images`

Unchanged from phase 0.

## Resources

For every cluster-scoped resource:
- `cluster_id` is **Required** (kept as-is from phase-0). The single exception
  is `cloudless_subnet`, which can derive it from the parent VPC.
- Async-lifecycle resources (everything except `cloudless_ssh_key`) use the
  shared `waitFor` poll loop with a 30-min default timeout.
- Read paths handle 404 and "removed terminal status" identically by removing
  the resource from state.

Below: only the changes from phase-0 status.

### `cloudless_ssh_key` (existing, unchanged)

No change. SSH keys aren't cluster-scoped.

### `cloudless_vpc` (existing, unchanged)

No schema change. `cluster_id` stays Required + ForceNew.

### `cloudless_subnet` (existing, schema change)

`cluster_id` becomes **Optional**. Resolution order:
1. Explicit `cluster_id` if supplied.
2. Otherwise, infer from `vpc_id` — read the VPC, take its `cluster_id`.
3. Otherwise, plan-time error ("set cluster_id explicitly or via vpc_id").

If both explicit `cluster_id` and the VPC's cluster are present and differ,
plan-time error (the API rejects this anyway, but we surface it earlier).

### `cloudless_vm` (existing, substantive update)

```hcl
resource "cloudless_vm" "app" {
  name             = "app"
  configuration_id = "..."
  # cluster_id     = "..."   # optional, falls back to provider default

  boot_disk {
    storage_id = cloudless_storage.boot.id   # OR inline-create fields
  }

  data_disk_ids = [cloudless_storage.x.id, cloudless_storage.y.id]
  ssh_key_ids   = [cloudless_ssh_key.me.id]
}

# Public IP and SG are bound via separate attachment resources — see below.
```

| Field | Change vs phase-0 | Update behavior |
|---|---|---|
| `cluster_id` | unchanged — Required | ForceNew |
| `boot_disk` block | unchanged shape — still supports `storage_id` reference OR inline-create fields, mutually exclusive within the block | ForceNew on any change (boot-disk swap recreates VM) |
| `data_disks` block | **REMOVED** | — |
| `data_disk_ids` (list of strings) | **NEW** | At create: in VM POST payload as `dataDisks`. At update: diff old vs new → call `/storages/add` and `/storages/remove`. |
| `public_ip` block | **REMOVED** | use `cloudless_vm_public_ip_attachment` |
| `ssh_key_ids` | unchanged | ForceNew (Fluence applies SSH keys at first boot) |
| `name` | unchanged | in-place via PATCH `/v2/vms/{id}` |

**Computed-only outputs** (not input attributes): `id`, `user_id`, `status`,
`boot_disk_id`, `network_interface_ids`, `public_ip_id` (read-only mirror of
whatever public IP is currently attached, populated from the API DTO),
`restart_required`, `created_at`, `updated_at`.

`data_disk_ids` is Required+Computed: the user supplies the list, and Read
populates state with whatever the API actually reports — so an out-of-band
`/storages/add` call shows up as drift.

**No `public_ip_id` or `security_group_id` input fields on the VM.** Both are
bound via separate attachment resources (`cloudless_vm_public_ip_attachment`
and `cloudless_security_group_attachment`). Rationale:
- The Fluence API may add multi-public-IP-per-VM later; an attachment resource
  doesn't need a breaking schema change to support it.
- SG is genuinely interface-scoped today (not VM-scoped), so a VM-level field
  would be a fiction.

**Migration**: phase-0 state doesn't carry the new fields. There's no
production data yet, so the schema change is breaking-but-acceptable.

### `cloudless_security_group` (NEW)

```hcl
resource "cloudless_security_group" "web" {
  cluster_id = "..."
  name       = "web"

  # Allow only the listed ingress rules; everything else dropped.
  ingress_mode = "allow_listed"
  ingress {
    protocol = "tcp"          # tcp | udp | icmp | all
    ports    = "443"          # all | "443" | "8000-8100"
    cidr     = "0.0.0.0/0"    # exactly one of cidr / security_group_id
    type     = "ipv4"         # ipv4 (default) | ipv6
  }

  # Forbid all outbound.
  egress_mode = "deny_all"
}
```

| Attribute | Kind | Notes |
|---|---|---|
| `cluster_id` | Required, ForceNew | |
| `name` | Required | in-place updatable |
| `ingress_mode` | Optional, default `"allow_all"` | `"allow_all"` / `"allow_listed"` / `"deny_all"` |
| `egress_mode` | Optional, default `"allow_all"` | same enum |
| `ingress {}` (block, 0..N) | Optional | required and non-empty when `ingress_mode = "allow_listed"`; forbidden otherwise |
| `egress {}` (block, 0..N) | Optional | same rule for egress |
| `ingress.protocol` / `egress.protocol` | Required | `tcp`, `udp`, `icmp`, `all` |
| `ingress.ports` / `egress.ports` | Optional | required for tcp/udp; forbidden for icmp/all |
| `ingress.cidr` / `egress.cidr` | Optional | exactly one of cidr / security_group_id |
| `ingress.security_group_id` / `egress.security_group_id` | Optional | exactly one of cidr / security_group_id |
| `ingress.type` / `egress.type` | Optional, default `"ipv4"` | `ipv4` or `ipv6`; validated against `cidr` shape |
| `id`, `vpc_id`, `user_id`, `status`, `attached_to`, `created_at` | Computed | from DTO |

**Plan-time validation**:
- `ingress_mode = "allow_listed"` ⇒ at least one `ingress {}` block must be present.
- `ingress_mode = "allow_all"` or `"deny_all"` ⇒ zero `ingress {}` blocks.
- Same rules for `egress_mode` and `egress {}`.

**Marshal rules** (per direction, applied independently for ingress and egress):
- `mode = "allow_all"` ⇒ `{ type: "allowAll" }`. Never `null` (avoids the
  "reset to AllowAll" path from the OpenAPI spec, which is semantically the
  same but spelled differently — we always send the explicit form).
- `mode = "deny_all"` ⇒ `{ type: "allow", rules: [] }`.
- `mode = "allow_listed"` ⇒ `{ type: "allow", rules: [...] }`.
- `protocol="all"` ⇒ `protocolKind: "all"` (bare string).
- `protocol="icmp"` ⇒ `protocolKind: "icmp"` (bare string).
- `protocol="tcp"` + `ports="all"` ⇒ `{ tcp: { ports: "all" } }`.
- `protocol="tcp"` + `ports="443"` ⇒ `{ tcp: { ports: { exact: { value: 443 } } } }`.
- `protocol="tcp"` + `ports="8000-8100"` ⇒ `{ tcp: { ports: { range: { min: 8000, max: 8100 } } } }`.
- `cidr="x"` ⇒ `remote: { address: "x" }`.
- `security_group_id="y"` ⇒ `remote: { securityGroup: "y" }`.

**Lifecycle**: async (`new`/`created` → `ready`; `removing` → `removed`). PATCH
covers `name` and rules in-place. Delete is `POST /v1/security_groups/delete`
followed by polling. Errors verbatim from the API if still attached.

### `cloudless_storage` (NEW)

```hcl
resource "cloudless_storage" "data" {
  name         = "data"
  storage_type = "NVME"
  volume_gb    = 200
  replicated   = false
  os_image     = "https://..."   # optional; presence ⇒ boot disk
}
```

| Attribute | Kind | Notes |
|---|---|---|
| `cluster_id` | Required, ForceNew | |
| `name` | Required | in-place updatable |
| `storage_type` | Required, ForceNew | only `"NVME"` accepted on create today |
| `volume_gb` | Required | in-place updatable via PATCH; flagged as potentially disruptive in docs |
| `replicated` | Required, ForceNew | |
| `os_image` | Optional, ForceNew | URL of qcow2/raw image |
| Computed | | `id`, `user_id`, `status`, `role` (BOOT/DATA), `attached_to`, `created_at`, etc. |

**Lifecycle**: async. PATCH for name + volume_gb. Delete via `POST
/v1/storages/delete`. Cannot delete a storage attached to a VM — surface API
error.

### `cloudless_public_ip` (NEW)

```hcl
resource "cloudless_public_ip" "edge" {
  name         = "edge"
  address_type = "V4"
}
```

| Attribute | Kind | Notes |
|---|---|---|
| `cluster_id` | Required, ForceNew | |
| `name` | Required | in-place updatable |
| `address_type` | Required, ForceNew | `"V4"` only on create; `"V6"` returnable on read |
| Computed | | `id`, `address` (the IP string), `status`, `attached_to` (VM ID or null), `created_at` |

**Lifecycle**: async. PATCH for name only. Delete via `POST /v1/public_ips/delete`.

### Storage attachment — **NOT a separate resource**

We considered a `cloudless_vm_storage_attachment` and dropped it during the
walkthrough. Storage relationships are modelled on the VM via `data_disk_ids`
with smart Update. Rationale: storage already supports N-per-VM in the API
today (no future-proofing argument); ID list with smart Update is shorter HCL
and avoids the AWS-style "two ways to do it" trap.

### `cloudless_vm_public_ip_attachment` (NEW, attachment)

```hcl
resource "cloudless_vm_public_ip_attachment" "edge" {
  vm_id        = cloudless_vm.app.id
  public_ip_id = cloudless_public_ip.edge.id
}
```

| Attribute | Kind |
|---|---|
| `vm_id` | Required, ForceNew |
| `public_ip_id` | Required, ForceNew |
| `id` (computed) | `<vm_id>:<public_ip_id>` |

Endpoints:
- Create → `POST /v2/vms/{vm_id}/public_ip/add` with `{ publicIpId }` body.
- Read → `GET /v2/vms?ids=<vm_id>`, confirm `publicIp == public_ip_id`. Removed
  from state if not.
- Delete → `POST /v2/vms/{vm_id}/public_ip/remove` (no body).
- Import via `<vm_id>:<public_ip_id>`.

**Spec quirk**: the `/remove` endpoint takes no body and removes whatever IP
is currently attached. If something else attached a different IP between
state-read and delete, our delete would remove that other IP. We document the
race in the user-facing docs and don't try to defend against it.

**Why this is a separate resource even though `cloudless_vm` could carry a
single `public_ip_id` today**: per the entity walkthrough, we expect Fluence
to add multi-public-IP support. An attachment resource lets us evolve to
multi without a breaking schema change.

### `cloudless_security_group_attachment` (NEW, attachment)

```hcl
resource "cloudless_security_group_attachment" "web" {
  network_interface_id = cloudless_vm.app.network_interface_ids[0]
  security_group_id    = cloudless_security_group.web.id
}
```

| Attribute | Kind |
|---|---|
| `network_interface_id` | Required, ForceNew |
| `security_group_id` | Required (in-place updatable: change ⇒ PATCH with new ID) |
| `id` (computed) | `network_interface_id` (UUID is globally unique) |
| `vm_id` (computed) | resolved at read time by listing interfaces; stored in state |

Create / Update → `PATCH /v2/vms/{vm_id}/interfaces/{interface_id}` with
`{ securityGroupId }`. Delete → same PATCH with `securityGroupId: null`.

The PATCH endpoint requires `vm_id`. We resolve it at create time (the user
supplied `network_interface_id` indirectly via `cloudless_vm.x.network_interface_ids[0]`,
but the attachment itself doesn't carry vm_id). Resolution: `GET /v2/vms` list
filtered to find the one containing this network interface ID, store as
computed `vm_id` in state. On import, accept `<vm_id>:<network_interface_id>`
as the import ID to skip the lookup.

**No convenience inline field on `cloudless_vm`.** Multi-interface VMs would
make any single-SG shortcut ambiguous; users compose explicitly.

## Validators (Phase 2 cross-cutting)

A small `internal/provider/validators.go` package with reusable validators:

| Validator | Applied to |
|---|---|
| `validUUID()` | every `cluster_id`, `vpc_id`, `*_id` field |
| `validCIDR(family="any"/"ipv4"/"ipv6")` | `subnet.ipv4_cidr`, `subnet.ipv6_cidr`, SG `cidr` |
| `validPortSpec()` | SG `ports` field — parses `"all"` / int / `min-max` |
| `validProtocol()` | SG `protocol` field — enum |
| `validIPType()` | SG `type` field — `ipv4`/`ipv6` |
| `validRegionCode()` | data source `region` filter — enforces 2-char uppercase |

These are added in Phase 2, not Phase 1, to avoid blocking the resource work
on validator design churn.

## Testing

### Unit tests — Phase 1 (TDD)

Every new resource lands with a unit test that drives the full lifecycle
(create → apply → re-apply → destroy) against an in-memory mock server.

The mock server (`internal/client/mock/server.go`) is a real `httptest.Server`
that holds an in-memory map of resources keyed by ID. It implements the API
surface the tests exercise:

- POST/PATCH/POST-delete paths for ssh_key, vpc, subnet, security_group,
  storage, public_ip
- POST `/v2/vms`, GET, PATCH, terminate, storages/add, storages/remove,
  public_ip/add, public_ip/remove, interfaces, interface PATCH
- GET `/v1/clusters`, `/v1/datacenters`, `/v1/configurations/virtual_machines`,
  `/v1/storages/default_images`

Async resources transition status on each subsequent GET (one tick = one
status step), so the poll loops actually exercise — but in a few hundred ms
total per test. Tests use `terraform-plugin-testing` v2's framework-style
`resource.UnitTest` runner pointed at the mock.

Test files for the *existing* phase-0 resources (ssh_key, vpc, subnet, vm) are
also part of Phase 2's hardening — we don't go back during Phase 1.

### Acceptance tests — Phase 2

Same `resource.Test` (real API) for each resource, gated on:

```go
if os.Getenv("TF_ACC") != "1" || os.Getenv("FLUENCE_API_KEY") == "" {
    t.Skip("set TF_ACC=1 and FLUENCE_API_KEY to run")
}
```

Run locally with `TF_ACC=1 FLUENCE_API_KEY=... go test ./internal/provider/
-run TestAcc -timeout 60m`. Never executed in CI (D4).

Each acc test creates resources with a `tf-acc-<random>-` name prefix and runs
a teardown best-effort cleanup in `t.Cleanup`. We document the pattern in the
README.

## Release pipeline (Phase 2 — configured, not triggered)

`.goreleaser.yml` ships:

- Build matrix: linux/darwin/windows × amd64/arm64
- GPG signing of artifacts
- SHA256 checksums
- Output named `terraform-provider-cloudless_v<version>` per registry

The Nix dev shell adds `goreleaser` and `gnupg`. No release is *triggered* in

## CI (Phase 2)

One GitHub Actions workflow, `.github/workflows/build.yml`:

- Triggers: pull_request, push to `main`
- Steps: checkout → setup-go (1.25) → `go build ./...` → `go vet ./...` → `go
  test ./...` (unit only — no `TF_ACC`)
- One job, no matrix. Single Go version (1.25 from the flake). OS: ubuntu-latest.

Acceptance tests do not run in CI (D4). `goreleaser` does not run in CI (no
release tag triggers in this spec).

## Phase split

### Phase 1 — new resources (TDD, unit tests only)

Plan items, in order:
1. Subnet `cluster_id`-from-VPC inference — Optional + derive-from-VPC logic
   in the subnet resource. Tests for the resolver.
2. Mock server (`internal/client/mock/`) — implements just enough API to drive
   the tests we write next.
3. New data sources: `cloudless_cluster` + enhanced `cloudless_clusters`.
   Country/city filter + datacenter join logic. Unit-tested via the mock.
4. `cloudless_security_group` resource.
5. `cloudless_storage` resource.
6. `cloudless_public_ip` resource.
7. `cloudless_vm` update: replace inline `data_disks` block with `data_disk_ids`
   list + smart Update; remove inline `public_ip` block; expose
   `network_interface_ids` computed.
8. `cloudless_vm_public_ip_attachment` resource.
9. `cloudless_security_group_attachment` resource.

Each item is a TDD cycle: write the test against the mock, fail, implement,
pass. Each resource lands with: schema, client method(s), unit test, examples
file (HCL only — docs come in Phase 2). Existing resources gain unit tests in
Phase 2.

### Phase 2 — cross-cutting hardening

Plan items, in order:
1. Validators package + apply to all resources.
2. Generate docs via `tfplugindocs`; commit `docs/` and `templates/`.
3. Acceptance tests for every resource (locally runnable).
4. Unit tests for the existing phase-0 resources (ssh_key, vpc, subnet, vm).
5. CI workflow (`.github/workflows/build.yml`).
6. Release pipeline (`.goreleaser.yml`, dev shell additions).
7. README rewrite covering the new resource surface, dev workflow, release
   process.

## Open items (intentional, deferred)

These are flagged but not addressed in this spec:

- **Module path / registry namespace.** `github.com/cloudless/terraform-provider-cloudless`
  is a placeholder. Renaming it is a one-shot find/replace; doesn't affect
  design.
- **First release tag.** Out of scope per goal; spec ships pipeline only.
- **CI-gated acceptance tests.** Per D4. Future spec when budget/secret
  handling is sorted.
- **Storage-resize UX safety.** `volume_gb` change is in-place per D11. If it
  proves disruptive in practice, a follow-up could add a `force_replace = true`
  toggle, but we don't pre-build for that.
- **Multi-network-interface VMs.** All current Fluence VMs have exactly one
  interface in practice. Our SG-attachment resource is per-interface, so
  multi-interface works — but the docs assume the common single-interface
  case.

## Glossary

- **Cluster** — Fluence's locality unit. UUID. ≈ AWS AZ in granularity, used
  like a region.
- **Datacenter (`dc_id`)** — Physical site. UUID. Has a `countryCode`,
  `cityCode`, `tier`, `certifications`, `slug`.
- **Region** (in this provider's data sources only) — synonym for ISO 3166-1
  alpha-2 country code (`"DE"`, `"PL"`, `"US"`, …). Pure filter attribute, no
  resource carries it.
